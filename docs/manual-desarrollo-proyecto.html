1. Qué es este proyecto

Este sistema es un portal de proveedores y compras para TotalGas. No es una app pequeña ni un demo de Laravel: maneja usuarios internos, proveedores, documentos, requisiciones, cotizaciones, aprobaciones, presupuestos, órdenes de compra, recepciones y algunas integraciones externas.

Resumen funcional El flujo principal es: Requisición → Cotización/RFQ → Aprobación → Orden de Compra → Recepción. En paralelo existe el portal del proveedor, catálogos, presupuestos y revisión documental.

3. Checklist del primer día

  1. Confirma que tienes PHP 8.2, Composer, Node y acceso a la base SQL Server correcta.
  2. Revisa el archivo .env y valida que la conexión de BD, mail y URL local sean las adecuadas.
  3. Instala dependencias: composer install y npm install.
  4. Genera llave de app si hace falta: php artisan key:generate.
  5. Levanta migraciones y seeders solo si tu entorno local lo requiere: php artisan migrate --seed.
  6. Arranca el entorno de desarrollo con composer dev o al menos con php artisan serve y npm run dev.
  7. Prueba iniciar sesión y revisar al menos: dashboard, catálogo, requisiciones y proveedor.
  8. Lee también la documentación existente: README.md y SISTEMA_FLUJO_ACTUAL.md.
No des por hecho que la documentación antigua está 100% alineada Este repositorio tiene partes genéricas y partes ya personalizadas. Antes de tocar algo importante, verifica la ruta, el controlador, el modelo y la migración reales del módulo.

4. Cómo está organizado

Ruta Qué hay ahí Cuándo entrar
routes/web.php Rutas web principales Siempre que quieras entender cómo entra un módulo.
routes/api.php Endpoints API Para integraciones como empleados desde Python.
app/Http/Controllers Controladores Para seguir la lógica de una pantalla o acción.
app/Http/Requests Form Requests Para ver o agregar validaciones formales.
app/Models Modelos Eloquent Para relaciones, scopes, casts, reglas de negocio.
app/Services Lógica de negocio compartida Cuando una acción ya es demasiado grande para el controlador.
app/Enum Enums de estados y catálogos Antes de inventar strings nuevos de estado.
resources/views Vistas Blade Para UI server-rendered y layouts.
app/Livewire y resources/views/livewire Componentes Livewire Cuando veas pantallas más dinámicas sin SPA completa.
database/migrations Estructura BD Si vas a tocar tablas, columnas, índices o pivots.
database/seeders Datos base Cuando agregues catálogos o dependencias iniciales.
tests/Feature Pruebas funcionales Para ver ejemplos y agregar cobertura.
Layout principal La mayoría de las vistas de negocio extienden resources/views/layouts/zircos.blade.php. Ese layout ya carga sidebar, navbar, DataTables, Select2, SweetAlert, estilos del template y stacks de @push('styles') / @push('scripts').
Layouts secundarios También existen layouts.app y layouts.guest por la parte de auth/Breeze. Si vas a crear una pantalla interna del sistema, casi siempre debes usar Zircos, no Breeze.

5. Flujo de trabajo diario recomendado

  1. Empieza por la ruta: localiza el endpoint en routes/web.php o routes/api.php.
  2. Abre el controlador y sigue qué vista, modelo, servicio o request usa.
  3. Revisa el modelo involucrado: relaciones, casts, scopes, enums, métodos de negocio.
  4. Si hay DataTable, busca el método datatable() o endpoint AJAX relacionado.
  5. Si hay formularios, revisa si usan un FormRequest.
  6. Antes de cambiar estados, revisa si el módulo ya tiene enum o reglas de transición.
  7. Antes de tocar visibilidad, revisa empresa, centro de costo y roles.
  8. Haz el cambio mínimo necesario y luego valida UI, rutas, permisos y BD.
Regla práctica En este proyecto rara vez basta con editar una sola vista. Normalmente una pantalla toca al menos: ruta + controlador + vista + validación + permisos + consulta.

6. Si tienes que crear una vista

Escenario A: solo una pantalla nueva informativa

  1. Define la ruta en routes/web.php.
  2. Crea o reutiliza un método en un controlador.
  3. Haz que ese método retorne view('modulo.nombre').
  4. Crea el archivo Blade en resources/views/modulo/nombre.blade.php.
  5. Extiende layouts.zircos si es una pantalla interna del sistema.
  6. Agrega título, breadcrumbs y contenido.
  7. Si debe aparecer en navegación, revisa el sidebar parcial.
  8. Protege la ruta con middleware o roles si aplica.

Escenario B: vista de formulario o CRUD

  1. Primero identifica si ya existe patrón del módulo: index/create/edit/show.
  2. Crea la ruta o usa Route::resource si el módulo es clásico.
  3. En el controlador prepara la data que necesita la vista: catálogos, selects, defaults.
  4. Usa FormRequest si el formulario guarda datos.
  5. En Blade usa @csrf y, para editar, @method('PUT') o el método correspondiente.
  6. Muestra errores con @error y conserva valores con old().
  7. Si dependes de DataTables, Select2 o SweetAlert, el layout ya trae mucho cargado.
  8. Si la pantalla tiene comportamiento complejo, empuja scripts con @push('scripts').

Plantilla mental para crear una vista nueva

// routes/web.php
Route::get('/mi-modulo/create', [MiModuloController::class, 'create'])->name('mi-modulo.create');
Route::post('/mi-modulo', [MiModuloController::class, 'store'])->name('mi-modulo.store');

// app/Http/Controllers/MiModuloController.php
public function create(): View
{
    return view('mi_modulo.create', [
        'companies' => Company::orderBy('name')->get(),
    ]);
}

// resources/views/mi_modulo/create.blade.php
@extends('layouts.zircos')
@section('title', 'Nuevo registro')
@section('page.title', 'Nuevo registro')
@section('content')
    <form method="POST" action="{{ route('mi-modulo.store') }}">
        @csrf
        ...
    </form>
@endsection
Antes de crear una vista desde cero Busca una parecida en resources/views. Este proyecto ya tiene muchos formularios reutilizables en carpetas como partials. Copiar el patrón correcto ahorra tiempo y reduce inconsistencias.

7. Si tienes que mover algo en BD

“Mover algo en la BD” puede significar varias cosas: agregar una columna, crear tabla, cambiar una relación, sembrar catálogo o ajustar lógica por datos. En Laravel eso normalmente pasa por migraciones, modelos y, a veces, seeders.

Reglas base

  1. No edites migraciones viejas si ese cambio ya pudo correr en otros entornos compartidos.
  2. Para cambios nuevos, crea una migración nueva.
  3. Después de tocar una tabla, revisa el modelo asociado.
  4. Si el cambio afecta formularios, actualiza validaciones y vistas.
  5. Si afecta datos base, revisa seeders.
  6. Si cambia estados o reglas, revisa enums y lógica de negocio.

Casos típicos

Necesidad Qué tocar normalmente
Agregar columna Migración nueva + modelo (fillable, casts) + request + vista.
Nueva tabla Migración + modelo + relaciones + controlador/vistas si tendrá UI.
Nueva relación pivot Migración + relaciones en modelos + lógica de guardado.
Nuevo catálogo base Migración o datos existentes + seeder + UI de administración si aplica.
Cambiar estado de negocio Columna/enum + reglas de transición + vistas + notificaciones + pruebas.

Pasos recomendados para un cambio de BD

  1. Localiza la tabla y su migración original para entender la estructura actual.
  2. Crea una migración nueva con un nombre claro.
  3. Corre la migración localmente.
  4. Actualiza el modelo: fillable, casts, relaciones, scopes o helpers.
  5. Actualiza validaciones en FormRequest o controlador.
  6. Actualiza vistas y cualquier DataTable o endpoint API.
  7. Si el cambio requiere datos iniciales, agrega o ajusta seeder.
  8. Prueba crear, editar, listar y consultar el dato.
php artisan make:migration add_mi_columna_to_mi_tabla_table
php artisan migrate
Mucho cuidado con SQL Server Algunos cambios que son triviales en MySQL no lo son igual aquí. Revisa tipos, constraints, defaults, índices y compatibilidad de expresiones SQL si haces consultas manuales.

8. Si tienes que crear un módulo o CRUD completo

  1. Define si el módulo es catálogo, flujo de negocio o soporte.
  2. Crea tabla y modelo si no existen.
  3. Crea controlador.
  4. Usa FormRequest para validaciones si habrá guardar/editar.
  5. Define rutas. Si encaja, usa Route::resource; si no, crea rutas explícitas.
  6. Crea vistas index/create/edit/show según necesidad.
  7. Si el listado será grande, agrega endpoint datatable y DataTables server-side.
  8. Agrega permisos/roles si el acceso no debe ser general.
  9. Si requiere aparecer en menú, ajusta el sidebar.
  10. Agrega pruebas mínimas.

Patrón que ya existe mucho aquí

index()
datatable()
create()
store()
show()
edit()
update()
destroy()

También es común que exista una carpeta partials con el formulario reutilizable entre create y edit.

9. Convenciones reales del proyecto

1. Layout interno = Zircos Para pantallas internas del sistema, lo normal es extender layouts.zircos.
2. Validación Ya existen varios FormRequest. Si un formulario es serio, úsalo en vez de meter reglas largas en el controlador.
3. Roles y permisos El proyecto usa Spatie. Antes de agregar una pantalla, piensa quién debe verla y qué middleware necesita.
4. Filtros por empresa/centro de costo No olvides que muchas entidades están acotadas por empresa y centro de costo asignados al usuario.
5. Estados Algunos módulos ya usan enums en app/Enum. Antes de meter strings nuevos, verifica si el estado ya existe allí.
6. DataTables Muchos listados no renderizan todo en Blade; usan endpoints JSON con Yajra DataTables.
7. Transacciones En operaciones que tocan varias tablas, el proyecto ya usa DB::transaction(). Mantén ese patrón.
8. Notificaciones Varias transiciones importantes disparan correos y notificaciones en base de datos. No rompas esos flujos al cambiar estados.

Qué revisar antes de cambiar lógica de negocio

  1. Modelo y sus métodos.
  2. Enums en app/Enum.
  3. Servicios en app/Services.
  4. Notificaciones en app/Notifications.
  5. Jobs o scheduler si el proceso es automático.
  6. Rutas y middleware.

10. Riesgos y cosas delicadas

1. Mezcla de capas nuevas y legado Hay piezas de Breeze/Tailwind, pantallas con Zircos/Bootstrap, componentes Livewire y pantallas Blade clásicas. No asumas un solo patrón global.
2. Visibilidad restringida Productos, requisiciones y presupuestos están ligados a empresa y centro de costo. Una query sin esos filtros puede mostrar o editar datos que no corresponden.
3. Los estados importan mucho Requisiciones, RFQ, órdenes, recepciones y productos tienen estados que gobiernan permisos y flujo. Cambiar una transición sin revisar el resto del flujo puede romper más de una pantalla.
4. Scheduler activo El scheduler tiene al menos una tarea para cerrar órdenes inactivas. Si tocas lógica de OC/OCD, revisa también routes/console.php y los comandos Artisan relacionados.
5. Documentos, correos y evidencias Hay módulos con archivos, notificaciones y PDFs. Cuando toques alguno, revisa storage, mails, jobs y vistas de correo.

11. Cómo validar tus cambios

Validación mínima técnica

  1. Corre el flujo manual en navegador.
  2. Confirma que la ruta responde y no rompe middleware o permisos.
  3. Si cambiaste BD, prueba crear y editar registros reales del módulo.
  4. Si cambiaste listados, revisa el DataTable y sus filtros.
  5. Si cambiaste estados, valida al menos la transición feliz y una inválida.
  6. Ejecuta pruebas existentes relevantes si las hay.
php artisan test
php artisan test --filter EmployeeCatalogTest
php artisan pint
Importante El proyecto sí tiene pruebas, pero no cubre todo. Que una suite pase no significa que todo el flujo de negocio esté sano. Aquí sigue siendo clave validar manualmente el módulo que tocaste.

12. Checklists rápidos

Checklist: crear una vista nueva

  • ¿La ruta ya existe o la tienes que crear?
  • ¿La pantalla debe usar layouts.zircos?
  • ¿Necesita permisos o middleware?
  • ¿Necesita datos desde controlador?
  • ¿Es un simple Blade o un listado con DataTable?
  • ¿Debe aparecer en sidebar o navegación?
  • ¿Probaste errores y valores con old()?

Checklist: tocar base de datos

  • ¿Debes crear migración nueva en vez de editar una vieja?
  • ¿Actualizaste modelo, casts, fillable y relaciones?
  • ¿Actualizaste validaciones y vistas?
  • ¿Afecta seeders o catálogos base?
  • ¿Afecta estados, permisos o notificaciones?
  • ¿Probaste la migración y el rollback localmente si aplica?

Checklist: antes de cerrar una tarea

  • La ruta carga correctamente.
  • El rol correcto sí puede entrar.
  • Un rol incorrecto no rompe o no entra.
  • El formulario valida bien.
  • Los datos se guardan en la tabla esperada.
  • La UI no quedó desalineada con Zircos/Bootstrap.
  • No rompiste sidebar, breadcrumbs ni DataTables.
  • Probaste al menos el caso feliz y un caso inválido.

13. Recomendación final para trabajar bien aquí

Si vienes empezando con Laravel, no intentes memorizar todo el framework antes de tocar código. Para este proyecto, lo más rentable es aprender a seguir el recorrido real de cada funcionalidad: ruta → controlador → request → modelo/servicio → vista. Con ese hábito vas a poder moverte mucho más rápido y con menos errores.

Forma más segura de pensar cada cambio “¿Qué pantalla toca esto?, ¿qué ruta entra?, ¿qué controlador lo resuelve?, ¿qué modelo o tabla afecta?, ¿qué rol lo usa?, ¿qué estado de negocio cambia?”
